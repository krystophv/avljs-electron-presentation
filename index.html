What is Electron?

https://www.wildnettechnologies.com/build-cross-platform-desktop-apps-with-electron/

Alright, now we're getting somewhere.
Electron, in a nutshell, is a Chromium browser and a Node.js runtime bundled together.
The Chromium browser is the window that you see and interact with.
The Node.js server is what manages the runtime.
The main process is where you manage the window and the renderer processes and the renderer processes are where you write your presentation logic.
Broadly speaking, the main process runs in Node.js   (think Puppeteer) and the renderer processes run in Chromium.
The main process and the renderer processes communicate with each other using IPC.
You _can_ opt out of the sandbox, but it's not recommended - you should use preload scripts to expose
the necessary APIs to the renderer processes.

Why use Electron?

Reasons to use Electron

Web technologies: Electron leverages web technologies, making it easy for web teams to build desktop applications using their existing skills.

Code reuse: Electron applications can reuse code from your web applications to build your desktop applications.

Rich ecosystem: NodeJS and Web JavaScript have vibrant and active communities, which means you can find a wide range of plugins, libraries, and resources available.

Native capabilities: Electron provides access to native APIs, allowing you to leverage the full power of the underlying operating system and integrate with system-level features.

Cross-platform compatibility: Electron allows you to build desktop applications that can run on Windows, macOS, and Linux without the need for platform-specific code.*

* generally

It's enough: Many desktop apps don't do a whole lot more than what a web app can do. (this is foreshadowing)

Do we need to go back to the slide with the list of Electron apps?
Big web applications already have a web team. Now you've got a desktop team.
You can generally reuse your web application code to build your desktop application.
Many desktop applications don't do a whole lot more than what a web app can do, and the lift of having the  separate Chromium and NodeJS runtimes isn't a strain for most modern computers. 
Optimizations are possible, like using canvas for rendering the terminal in VSCode.

Continuous Integration

Continuous Integration (CI) servers automate the process of building, testing, and deploying software.

Also known as build servers for the sake of this presentation. CI servers can be used to build Electron Forge applications across platforms and publish them to destinations like GitHub.

Github Actions

.github\workflows\build.yml

name: Build
on: [push, pull_request]

jobs:
build_on_linux:
  runs-on: ubuntu-latest
  steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
    with:
      node-version: 20
      cache: 'yarn'
  - name: install dependencies
    run: yarn install --frozen-lockfile
  - name: build
    run: yarn make

build_on_mac:
  runs-on: macos-latest
  steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-node@v4
    with:
      node-version: 20
      cache: 'yarn'
  - name: install dependencies
    run: yarn install --frozen-lockfile
  - name: build
    run: yarn make

build_on_win:
  runs-on: windows-latest

  - uses: actions/checkout@v4

This is the Github Actions workflow file.
It's telling Github Actions to run the build job on push and pull requests.

Code Signing

Code signing is the process of digitally signing your application's executable files to verify their authenticity and integrity.

It helps users trust that the application comes from a known source and hasn't been tampered with.

You need to do it.

You need code signing certificates, they're not free.

Setting Github Actions runner permissions

In order for the Github publisher to run successfully, you need to set the necessary permissions for the Github Actions runner.

This is the Github Actions workflow permissions page.
You need to set the write permission for the Github Actions runner to be able to create the draft release.

Add publish action

.github\workflows\release.yml

name: Release
on:
  push:
    tags:
      - 'v*'

jobs:
  build:
    strategy:
      matrix:
        os:
          [
            { name: 'linux', image: 'ubuntu-latest' },
            { name: 'windows', image: 'windows-latest' },
            { name: 'macos', image: 'macos-latest' },
          ]
    runs-on: ${{ matrix.os.image }}
    steps:
      - name: Github checkout
        uses: actions/checkout@v4
      - name: Use Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'yarn'
      - name: install dependencies
        run: yarn install --frozen-lockfile
      - name: publish
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: yarn run publish
This is the Github Actions release workflow file.
It's telling Github Actions to run the release job on tag push with a version number.
It's also telling Github Actions to run the release job on linux, macos, and windows.

Push a tag

After configuring the publisher and providing the necessary permissions, you can push a version tag to run the publish action.

This will create a draft release on your Github repository.

Advancements in PWAs

Progressive Web Apps (PWAs) have made significant advancements in recent years, offering capabilities that were once exclusive to frameworks like Electron.

  • Offline support
  • Push notifications
  • Background sync
  • Access to device features: camera, microphone, and geolocation

Advancements in PWAs
